71// KeySegmentsExtractor - A class for splitting a key into meaningful pieces, or
72// "segments" for filtering purposes. We say the first key segment has segment
73// ordinal 0, the second has segment ordinal 1, etc. To simplify satisfying some
74// filtering requirements, the segments must encompass a complete key prefix (or
75// the whole key). There cannot be gaps between segments (though segments are
76// allowed to be essentially unused), and segments cannot overlap.
77//
78// Keys can also be put in "categories" to simplify some configuration and
79// handling. A "legal" key or bound is one that does not return an error (as a
80// special, unused category) from the extractor. It is also allowed for all
81// keys in a category to return an empty sequence of segments.
82//
83// To eliminate a confusing distinction between a segment that is empty vs.
84// "not present" for a particular key, each key is logically assiciated with
85// an infinite sequence of segments, including some infinite tail of 0-length
86// segments. In practice, we only represent a finite sequence that (at least)
87// covers the non-trivial segments.
88//
89// Once in production, the behavior associated with a particular GetId()
90// cannot change. Introduce a new GetId() when introducing new behaviors.
91// See also SstQueryFilterConfigsManager below.
92//
93// This feature hasn't yet been validated with user timestamp.
94//
95// = A SIMPLIFIED MODEL =
96// Let us start with the easiest set of contraints to satisfy with a key
97// segments extractor that generally allows for correct point and range
98// filtering, and add complexity from there. Here we first assume
99// * The column family is using the byte-wise comparator, or reverse byte-wise
100// * A single category is assigned to all keys (by the extractor)
101// * Using simplified criteria for legal segment extraction, the "segment
102// maximal prefix property"
103//
104// SEGMENT MAXIMAL PREFIX PROPERTY: The segment that a byte is assigned to can
105// only depend on the bytes that come before it, not on the byte itself nor
106// anything later including the full length of the key or bound.
107//
108// Equivalently, two keys or bounds must agree on the segment assignment of
109// position i if the two keys share a common byte-wise prefix up to at least
110// position i - 1 (and i is within bounds of both keys).
111//
112// This specifically excludes "all or nothing" segments where it is only
113// included if it reaches a particular width or delimiter. A segment resembling
114// the FixedPrefixTransform would be illegal (without other assumptions); it
115// must be like CappedPrefixTransform.
116//
117// This basically matches the notion of parsing prefix codes (see
118// https://en.wikipedia.org/wiki/Prefix_code) except we have to include any
119// partial segment (code word) at the end whenever an extension to that key
120// might produce a full segment. An example would be parsing UTF-8 into
121// segments corresponding to encoded code points, where any incomplete code
122// at the end must be part of a trailing segment. Note a three-way
123// correspondence between
124// (a) byte-wise ordering of encoded code points, e.g.
125// { D0 98 E2 82 AC }
126// { E2 82 AC D0 98 }
127// (b) lexicographic-then-byte-wise ordering of segments that are each an
128// encoded code point, e.g.
129// {{ D0 98 } { E2 82 AC }}
130// {{ E2 82 AC } { D0 98 }}
131// and (c) lexicographic ordering of the decoded code points, e.g.
132// { U+0418 U+20AC }
133// { U+20AC U+0418 }
134// The correspondence between (a) and (b) is a result of the segment maximal
135// prefix property and is critical for correct application of filters to
136// range queries. The correspondence with (c) is a handy attribute of UTF-8
137// (with no over-long encodings) and might be useful to the application.
138//
139// Example types of key segments that can be freely mixed in any order:
140// * Capped number of bytes or codewords. The number cap for the segment
141// could be the same for all keys or encoded earlier in the key.
142// * Up to *and including* a delimiter byte or codeword.
143// * Any/all remaining bytes to the end of the key, though this implies all
144// subsequent segments will be empty.
145// As part of the segment maximal prefix property, if the segments do not
146// extend to the end of the key, that must be implied by the bytes that are
147// in segments, NOT because the potential contents of a segment were considered
148// incomplete.
149//
150// For example, keys might consist of
151// * Segment 0: Any sequence of bytes up to and including the first ':'
152// character, or the whole key if no ':' is present.
153// * Segment 1: The next four bytes, or less if we reach end of key.
154// * Segment 2: An unsigned byte indicating the number of additional bytes in
155// the segment, and then that many bytes (or less up to the end of the key).
156// * Segment 3: Any/all remaining bytes in the key
157//
158// For an example of what can go wrong, consider using '4' as a delimiter
159// but not including it with the segment leading up to it. Suppose we have
160// these keys and corresponding first segments:
161// "123456" -> "123" (in file 1)
162// "124536" -> "12" (in file 2)
163// "125436" -> "125" (in file 1)
164// Notice how byte-wise comparator ordering of the segments does not follow
165// the ordering of the keys. This means we cannot safely use a filter with
166// a range of segment values for filtering key range queries. For example,
167// we might get a range query for ["123", "125Z") and miss that key "124536"
168// in file 2 is in range because its first segment "12" is out of the range
169// of the first segments on the bounds, "123" and "125". We cannot even safely
170// use this for prefix-like range querying with a Bloom filter on the segments.
171// For a query ["12", "124Z"), segment "12" would likely not match the Bloom
172// filter in file 1 and miss "123456".
173//
174// CATEGORIES: The KeySegmentsExtractor is allowed to place keys in categories
175// so that different parts of the key space can use different filtering
176// strategies. The following property is generally recommended for safe filter
177// applicability
178// * CATEGORY CONTIGUOUSNESS PROPERTY: each category is contiguous in
179// comparator order. In other words, any key between two keys of category c
180// must also be in category c.
181// An alternative to categories when distinct kinds of keys are interspersed
182// is to leave some segments empty when they do not apply to that key.
183// Filters are generally set up to handle an empty segment specially so that
184// it doesn't interfere with tracking accurate ranges on non-empty occurrences
185// of the segment.
186//
187// = BEYOND THE SIMPLIFIED MODEL =
188//
189// DETAILED GENERAL REQUIREMENTS (incl OTHER COMPARATORS): The exact
190// requirements on a key segments extractor depend on whether and how we use
191// filters to answer queries that they cannot answer directly. To understand
192// this, we describe
193// (A) the types of filters in terms of data they represent and can directly
194// answer queries about,
195// (B) the types of read queries that we want to use filters for, and
196// (C) the assumptions that need to be satisfied to connect those two.
197//
198// TYPES OF FILTERS: Although not exhaustive, here are some useful categories
199// of filter data:
200// * Equivalence class filtering - Represents or over-approximates a set of
201// equivalence classes on keys. The size of the representation is roughly
202// proportional to the number of equivalence classes added. Bloom and ribbon
203// filters are examples.
204// * Order-based filtering - Represents one or more subranges of a key space or
205// key segment space. A filter query only requires application of the CF
206// comparator. The size of the representation is roughly proportional to the
207// number of subranges and to the key or segment size. For example, we call a
208// simple filter representing a minimum and a maximum value for a segment a
209// min-max filter.
210//
211// TYPES OF READ QUERIES and their DIRECT FILTERS:
212// * Point query - Whether there {definitely isn't, might be} an entry for a
213// particular key in an SST file (or partition, etc.).
214// The DIRECT FILTER for a point query is an equivalence class filter on the
215// whole key.
216// * Range query - Whether there {definitely isn't, might be} any entries
217// within a lower and upper key bound, in an SST file (or partition, etc.).
218// NOTE: For this disucssion, we ignore the detail of inclusive vs.
219// exclusive bounds by assuming a generalized notion of "bound" (vs. key)
220// that conveniently represents spaces between keys. For details, see
221// https://github.com/facebook/rocksdb/pull/11434
222// The DIRECT FILTER for a range query is an order-based filter on the whole
223// key (non-empty intersection of bounds/keys). Simple minimum and maximum
224// keys for each SST file are automatically provided by metadata and used in
225// the read path for filtering (as well as binary search indexing).
226// PARTITIONING NOTE: SST metadata partitions do not have recorded minimum
227// and maximum keys, so require some special handling for range query
228// filtering. See https://github.com/facebook/rocksdb/pull/12872 etc.
229// * Where clauses - Additional constraints that can be put on range queries.
230// Specifically, a where clause is a tuple <i,j,c,b1,b2> representing that the
231// concatenated sequence of segments from i to j (inclusive) compares between
232// b1 and b2 according to comparator c.
233// EXAMPLE: To represent that segment of ordinal i is equal to s, that would
234// be <i,i,bytewise_comparator,before(s),after(s)>.
235// NOTE: To represent something like segment has a particular prefix, you
236// would need to split the key into more segments appropriately. There is
237// little loss of generality because we can combine adjacent segments for
238// specifying where clauses and implementing filters.
239// The DIRECT FILTER for a where clause is an order-based filter on the same
240// sequence of segments and comparator (non-empty intersection of bounds/keys),
241// or in the special case of an equality clause (see example), an equivalence
242// class filter on the sequence of segments.
243//
244// GENERALIZING FILTERS (INDIRECT):
245// * Point queries can utilize essentially any kind of filter by extracting
246// applicable segments of the query key (if not using whole key) and querying
247// the corresponding equivalence class or trivial range.
248// NOTE: There is NO requirement e.g. that the comparator used by the filter
249// match the CF key comparator or similar. The extractor simply needs to be
250// a pure function that does not return "out of bounds" segments.
251// FOR EXAMPLE, a min-max filter on the 4th segment of keys can also be
252// used for filtering point queries (Get/MultiGet) and could be as
253// effective and much more space efficient than a Bloom filter, depending
254// on the workload.
255//
256// Beyond point queries, we generally expect the key comparator to be a
257// lexicographic / big endian ordering at a high level (or the reverse of that
258// ordering), while each segment can use an arbitrary comparator.
259// FOR EXAMPLE, with a custom key comparator and segments extractor,
260// segment 0 could be a 4-byte unsigned little-endian integer,
261// segment 1 could be an 8-byte signed big-endian integer. This framework
262// requires segment 0 to come before segment 1 in the key and to take
263// precedence in key ordering (i.e. segment 1 order is only consulted when
264// keys are equal in segment 0).
265//
266// * Equivalence class filters can apply to range queries under conditions
267// resembling legacy prefix filtering (prefix_extractor). An equivalence class
268// filter on segments i through j and category set s is applicable to a range
269// query from lb to ub if
270// * All segments through j extracted from lb and ub are equal.
271// NOTE: being in the same filtering equivalence class is insufficient, as
272// that could be unrelated inputs with a hash collision. Here we are
273// omitting details that would formally accommodate comparators in which
274// different bytes can be considered equal.
275// * The categories of lb and ub are in the category set s.
276// * COMMON SEGMENT PREFIX PROPERTY (for all x, y, z; params j, s): if
277// * Keys x and z have equal segments up through ordinal j, and
278// * Keys x and z are in categories in category set s, and
279// * Key y is ordered x < y < z according to the CF comparator,
280// then both
281// * Key y has equal segments up through ordinal j (compared to x and z)
282// * Key y is in a category in category set s
283// (This is implied by the SEGMENT MAXIMAL PREFIX PROPERTY in the simplified
284// model.)
285//
286// * Order-based filters on segments (rather than whole key) can apply to range
287// queries (with "whole key" bounds). Specifically, an order-based filter on
288// segments i through j and category set s is applicable to a range query from
289// lb to ub if
290// * All segments through i-1 extracted from lb and ub are equal
291// * The categories of lb and ub are in the category set s.
292// * SEGMENT ORDERING PROPERTY for ordinal i through j, segments
293// comparator c, category set s, for all x, y, and z: if
294// * Keys x and z have equal segments up through ordinal i-1, and
295// * Keys x and z are in categories in category set s, and
296// * Key y is ordered x < y < z according to the CF comparator,
297// then both
298// * The common segment prefix property is satisifed through ordinal i-1
299// and with category set s
300// * x_i..j <= y_i..j <= z_i..j according to segment comparator c, where
301// x_i..j is the concatenation of segments i through j of key x (etc.).
302// (This is implied by the SEGMENT MAXIMAL PREFIX PROPERTY in the simplified
303// model.)
304//
305// INTERESTING EXAMPLES:
306// Consider a segment encoding called BadVarInt1 in which a byte with
307// highest-order bit 1 means "start a new segment". Also consider BadVarInt0
308// which starts a new segment on highest-order bit 0.
309//
310// Configuration: bytewise comp, BadVarInt1 format for segments 0-3 with
311// segment 3 also continuing to the end of the key
312// x = 0x 20 21|82 23|||
313// y = 0x 20 21|82 23 24|85||
314// z = 0x 20 21|82 23|84 25||
315//
316// For i=j=1, this set of keys violate the common segment prefix property and
317// segment ordering property, so can lead to incorrect equivalence class
318// filtering or order-based filtering.
319//
320// Suppose we modify the configuration so that "short" keys (empty in segment
321// 2) are placed in an unfiltered category. In that case, x above doesn't meet
322// the precondition for being limited by segment properties. Consider these
323// keys instead:
324// x = 0x 20 21|82 23 24|85||
325// y = 0x 20 21|82 23 24|85 26|87|
326// z = 0x 20 21|82 23 24|85|86|
327// m = 0x 20 21|82 23 25|85|86|
328// n = 0x 20 21|82 23|84 25||
329//
330// Although segment 1 values might be out of order with key order,
331// re-categorizing the short keys has allowed satisfying the common segment
332// prefix property with j=1 (and with j=0), so we can use equivalence class
333// filters on segment 1, or 0, or 0 to 1. However, violation of the segment
334// ordering property on i=j=1 (see z, m, n) means we can't use order-based.
335//
336// p = 0x 20 21|82 23|84 25 26||
337// q = 0x 20 21|82 23|84 25|86|
338//
339// But keys can still be short from segment 2 to 3, and thus we are violating
340// the common segment prefix property for segment 2 (see n, p, q).
341//
342// Configuration: bytewise comp, BadVarInt0 format for segments 0-3 with
343// segment 3 also continuing to the end of the key. No short key category.
344// x = 0x 80 81|22 83|||
345// y = 0x 80 81|22 83|24 85||
346// z = 0x 80 81|22 83 84|25||
347// m = 0x 80 82|22 83|||
348// n = 0x 80 83|22 84|24 85||
349//
350// Even though this violates the segment maximal prefix property of the
351// simplified model, the common segment prefix property and segment ordering
352// property are satisfied for the various segment ordinals. In broader terms,
353// the usual rule of the delimiter going with the segment before it can be
354// violated if every byte value below some threshold starts a segment. (This
355// has not been formally verified and is not recommended.)
356//
357// Suppose that we are paranoid, however, and decide to place short keys
358// (empty in segment 2) into an unfiltered category. This is potentially a
359// dangerous decision because loss of continuity at least affects the
360// ability to filter on segment 0 (common segment prefix property violated
361// with i=j=0; see z, m, n; m not in category set). Thus, excluding short keys
362// with categories is not a recommended solution either.
363class KeySegmentsExtractor {
364 public:
365 // The extractor assigns keys to categories so that it is easier to
366 // combine distinct (though disjoint) key representations within a single
367 // column family while applying different or overlapping filtering
368 // configurations to the categories.
369 // To enable fast set representation, the user is allowed up to 64
370 // categories for assigning to keys with the extractor. The user will
371 // likely cast to their own enum type or scalars.
372 enum KeyCategory : uint_fast8_t {
373 kDefaultCategory = 0,
374 kMinCategory = kDefaultCategory,
375 // ... (user categories)
376 // Can be used for keys ordered before typical keys. Not necessarily an
377 // error.
378 kReservedLowCategory = 62,
379 // Can be used for keys ordered after typical keys. Not necessarily an
380 // error.
381 kReservedHighCategory = 63,
382 kMaxUsableCategory = kReservedHighCategory,
383
384 // Signals to the caller that an unexpected input or condition has
385 // been reached and filter construction should be aborted.
386 kErrorCategoryFilterScope = UINT8_MAX - 2,
387 kMinErrorCategory = kErrorCategoryFilterScope,
388 // Signals to the caller that an unexpected input or condition has
389 // been reached and SST construction (and compaction or flush)
390 // should be aborted.
391 kErrorCategoryFileScope = UINT8_MAX - 1,
392 // Signals to the caller that an unexpected input or condition has
393 // been reached and the DB should be considered to have reached an
394 // invalid state, at least in memory.
395 kErrorCategoryDbScope = UINT8_MAX,
396 };
397 using KeyCategorySet = SmallEnumSet<KeyCategory, kMaxUsableCategory>;
398
399 // The extractor can process three kinds of key-like inputs
400 enum KeyKind {
401 // User key, not including user timestamp
402 kFullUserKey,
403 // An iterator lower bound (inclusive). This should generally be handled
404 // the same as a full user key but the distinction might be useful for
405 // diagnostics or assertions.
406 kInclusiveLowerBound,
407 // An iterator upper bound (exclusive). Upper bounds are frequently
408 // constructed by incrementing the last byte of a key prefix, and this can
409 // affect what should be considered as a segment delimiter.
410 kExclusiveUpperBound,
411 };
412
413 // The extractor result
414 struct Result {
415 // Positions in the key (or bound) that represent boundaries
416 // between segments, or the exclusive end of each segment. For example, if
417 // the key is "abc|123|xyz" then following the guidance of including
418 // delimiters with the preceding segment, segment_ends would be {4, 8, 11},
419 // representing segments "abc|" "123|" and "xyz". Empty segments are
420 // naturally represented with repeated values, as in {4, 8, 8} for
421 // "abc|123|", though {4, 8} would be logically equivalent because an
422 // infinite sequence of 0-length segments is assumed after what is
423 // explicitly represented here. However, segments might not reach the end
424 // the key (no automatic last segment to the end of the key) and that is
425 // OK for the WEAK ordering property.
426 //
427 // The first segment automatically starts at key position 0. The only way
428 // to put gaps between segments of interest is to assign those gaps to
429 // numbered segments, which can be left unused.
430 std::vector<uint32_t> segment_ends;
431
432 // A category to assign to the key or bound. This default may be kept,
433 // such as to put all keys into a single category.
434 // IMPORTANT CURRENT LIMITATION from above: each category must be
435 // contiguous in key comparator order, so any key between two keys in
436 // category c must also be in category c. (Typically the category will be
437 // determined by segment 0 in some way, often the first byte.) The enum
438 // scalar values do not need to be related to key order.
439 KeyCategory category = kDefaultCategory;
440
441 void Reset() {
442 segment_ends.clear();
443 category = kDefaultCategory;
444 }
445 };
446
447 virtual ~KeySegmentsExtractor() {}
448
449 // A class name for this extractor. See also expectations in GetId().
450 virtual const char* Name() const = 0;
451
452 // An identifying string that is permanently associated with the behavior
453 // of this extractor. If a behavior change is made or set in the constructor,
454 // the id must change to avoid incorrect filtering behavior on DBs using a
455 // previous version of the extractor.
456 virtual std::string GetId() const {
457 // The default implementation assumes no configuration variance in the
458 // constructor and just returns the class name.
459 return Name();
460 }
461
462 // Populates the extraction result and returns OK. Error can be signaled
463 // with `kError` pseudo-categories. This function is expected to generate
464 // non-error results (though possibly empty) on all keys or bounds expected
465 // to be encountered by the DB. RocksDB will always call the function with
466 // a (pointer to a) default-initialized result object.
467 virtual void Extract(const Slice& key_or_bound, KeyKind kind,
468 Result* result) const = 0;
469};

553// Represents a complete strategy for representing filters in SST files
554// and applying them to optimize range and point queries by excluding
555// irrelevant SST files (as best we can). This is a set of filtering
556// schemes and a KeySegmentsExtractor. For performance, a single extractor
557// should be implemented to meet all the filtering needs of any given
558// column family. KeySegmentsExtractor and FilterInput should be flexible
559// enough that there is no loss of generality, e.g. with leaving segments

571// SstQueryFilterConfigsManager provides facilities for safe and effective
572// filtering version management, with simple dynamic upgrade/downgrade
573// support. It is designed to encourage a development pattern that
574// minimizes the risk of filter and extractor versioning bugs.
575//
576// SstQueryFilterConfigsManager is essentially an immutable mapping
577// from {config_name_string, version_number} -> SstQueryFilterConfigs
578// for some contiguous range of version numbers. It is also a starting
579// point for specifying which configuration should be used, with awareness
580// of other configurations that might already be persisted in SST files
581// or switched to dynamically.
582//
583// Background: a single codebase might have many use cases and for
584// each use case, a sequence of past, current, and future filtering
585// configurations. It is common for future configurations to be
586// implemented before automatically deployed in order to ensure that
587// a DB can be effectively opened and operated on by a recent older code
588// version. And it is common to maintain a reasonable history of past
589// configurations to ensure smooth upgrade path and proper handling of
590// older SST files that haven't been compacted recently.
591//
592// It would be possible to make SstQueryFilterConfigs dynamically
593// configurable through strings, but that would encourage deployment
594// of ad-hoc, under-tested configurations.
595//
596// Solution: the {config_name_string, version_number} -> SstQueryFilterConfigs
597// mapping in SstQueryFilterConfigsManager formalizes the history (past and
598// future) of approved/tested configurations for a given use case. Filter
599// configurations are kept paired with extractors that they make sense with.
600//
601// The version numbers are "global" to the SstQueryFilterConfigsManager so
602// that it is simple to determine whether a particular code version supports
603// a particular filtering version, regardless of which use case. Numbering
604// always starts with 1, as 0 is reserved for selecting a "no filters"
605// configuration
606//
607// Consider an example initialized with this Data:
608//
609// SstQueryFilterConfigsManager::Data data = {
610// {1, {{"foo", foo_configs}}},
611// {2, {{"bar", bar_configs},
612// {"baz", baz_configs}}},
613// {3, {{"bar", bar_configs_v2}}},
614// };
615//
616// For example, MakeSharedFactory(..., "baz", 1) will use a default empty
617// config, while both MakeSharedFactory(..., "baz", 2) and
618// MakeSharedFactory(..., "baz", 3) select `baz_configs`. Selecting version
619// >= 4 is rejected because those configurations are not known to this
620// version of the code.
621//
622// For correct operation, existing versions should be treated as immutable
623// (once committed to where they could enter production). For example, an
624// update to "baz" should be done in a new version (4), not by amending to
625// version 3. Note also (from before) that the behavior of named extractors
626// must not change, so changes to key segment extraction should introduce
627// new named extractors while keeping the old in the older configs.
628//
629// It is possible to eventually remove the oldest versions, as long as
630// * You won't be rolling back to that version.
631// * You don't have any SST files using an extractor that is only available
632// in that version (and prior).
633// * For each use case and version you might roll back to, you aren't removing
634// the configuration in effect for that version. In our example above, we
635// cannot simply remove version 1 because that would change the configuration
636// of "foo" at version 2. Moving {"foo", foo_configs} to version 2 would be
637// an acceptable work-around for retiring version 1.
638// * There are no gaps in the version numbers specified. Even if you completely
639// retire a use case and want to remove relevant code, you still need to keep
640// an explicit mapping, even if it's empty, as in `{3, {}}` if retiring "bar".
641//
642// Internally, the SstQueryFilterConfigsManager greatly simplifies lifetime
643// management for relevant objects such as KeySegmentExtractors, and provides
644// a lighter weight (and less troublesome) mechanism for relevant named object
645// look-up vs. ObjectRegistry. If following the guidelines above, any extractor
646// referenced in a read SST file should also be referenced by the
647// SstQueryFilterConfigsManager.
648//
649// Not user extensible
650class SstQueryFilterConfigsManager
651 : public std::enable_shared_from_this<SstQueryFilterConfigsManager> {
652 public:
653 using FilteringVersion = uint32_t;
654 using NamedConfigs = std::pair<std::string, SstQueryFilterConfigs>;
655 using Data =
656 std::vector<std::pair<FilteringVersion, std::vector<NamedConfigs>>>;
657
658 static Status MakeShared(const Data& data,
659 std::shared_ptr<SstQueryFilterConfigsManager>* out);
660
661 virtual ~SstQueryFilterConfigsManager() {}
